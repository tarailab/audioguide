## Verdict
The core journey experience looks **genuinely good** — the map-first HUD, the clear
single "▶ Play" call-to-action, and the polished Preferences screen are well above
"vibe-coded" quality. The main real items are around the **public-facing backend**:
the paid LLM endpoint has no rate limit, and trip endpoints trust a client header.

**Professionalism score: 8/10** (journey + prefs are strong; mainly missing error
states and the pre-launch hardening below).

## Top fixes (do these first)
1. **[Retracted — test bug, not an app bug]** ~~Trip Planner renders blank.~~ This was a
   wrong API mock in the test fixture, since fixed. The planner works in the real app.
2. **[High · security/cost]** `/api/story` calls Claude/OpenAI with no auth or rate
   limit. Public + unthrottled = anyone can run up your API bill. **Fix:** add a simple
   per-IP rate limit and a hard daily cap before public launch. (`backend/src/routes/story.js`)
3. **[High · security]** Trip endpoints trust a client-supplied `x-user-id` header
   (`frontend/src/services/api.js`). Anyone can read/modify another user's trips by
   changing the header. **Fix:** derive identity server-side (even a signed cookie) — don't trust the header.
4. **[Major · usability]** No visible **error state** when story generation or POI fetch
   fails — failures are thrown but the UI just shows nothing happening. **Fix:** surface a
   short "Couldn't load — tap to retry" message in the bottom card.
5. **[Minor · usability]** The GPS "Waiting for GPS…" idle state is good, but there's no
   **empty state** copy when GPS works yet no places are found beyond "No places yet" —
   consider a one-line hint ("Try widening interests in ⚙").
6. **[Polish]** The reload (↻) and auto-play (🔁/✋) controls rely on emoji as load-bearing
   icons. Fine for a prototype; for a public build, a consistent icon set reads more
   credibly (see design rubric).

## Usability findings
- **Journey (good):** clear hierarchy — map background, compact top HUD, one primary
  action. "2 places ahead" + tappable nearby chips is an intuitive model. ✅
- **Preferences (good):** consistent pill/segmented-button system, sensible grouping,
  helpful hint text under Voice. Single-choice vs multi-choice is visually distinguishable. ✅
- **Planner:** renders fine in the real app (the blank capture was a test-mock bug, now fixed).
- **Missing states:** loading is handled inline ("…"/"preparing…") but error states are
  absent across the app. This is the most common "amateur" tell and worth a pass.

## Design & architecture findings
- Visual consistency is strong: one dark theme, a coherent accent blue, consistent
  spacing and rounded controls. This does **not** look amateur. ✅
- `JourneyScreen.jsx` is doing a lot (GPS, compass, queue, admin panel, transport
  controls, map). It's readable but trending toward a god-component — consider extracting
  the bottom "now playing" card and the nearby strip into their own components.
- Good separation in `services/` (api/tts) and `store/`. Naming is clear throughout.
- Version badge + build info in the corner is a nice professional touch for testing.

## Security findings
- **High:** unauthenticated, unthrottled paid LLM endpoint (Top fix #2).
- **High:** `x-user-id` trust on trip endpoints (Top fix #3).
- **Medium:** confirm POI `image` URLs (from Wikidata/Wikipedia) are rendered only as
  `<img src>` (they are) and never injected as HTML — looks fine, keep it that way.
- **To verify:** the backend enriches via external services (Overpass/Wikidata) using
  client-provided coordinates — bound/validate them to avoid abusive query patterns.
- No hardcoded secrets seen in the frontend. (Backend `.env` is git-ignored — good.)

## What I checked and found OK
Journey & Preferences layout and flows; visual consistency and theming; frontend secret
hygiene; service-layer separation; the functional suite (8/8). The blank planner and the
public-endpoint exposure are the items that actually need action before launch.
